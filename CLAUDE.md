# CLAUDE.md — Substrate 引擎开发模式

你在 **Substrate** 开源引擎仓库里工作。**这是开发模式：你要真正实现这个引擎**（不是"只设计不落地"）。

> Substrate = 个人 AI agent 舰队的「共享状态层」引擎：git 原生、可被 agent 操作、可审计、自描述、可安全升级。
> 它是别人 fork 来搭建自己实例的**模板 + 机制 + 参考 skill + adapter + 迁移**。

## 先读这个

**开发依据 = `docs/BUILD-PLAN.md`**（完整设计 + 开发路线 P0–P5）。动手前通读它，按 phase 推进。
设计原理见 `docs/architecture.md`、`docs/concepts.md`；契约在 `schemas/`。

## 🔴 本仓库的红线（开源公开仓库）

1. **绝不含任何个人信息**：没有真实人名、具体机器/网络、密钥、私有路径、某个用户的偏好。引擎对所有用户中立。
2. **引擎不得依赖任何用户的偶然事实**（目录命名约定以外）。Engine / Instance 必须可分离：引擎只有机制，用户内容是另一个仓库。
3. **通用命名**：共享记忆槽位是 `memory/about-owner/`（主人名写在内容里，不进 folder 名）；实例名用 `<your-instance>` 占位，别硬编码。
4. 提交信息、示例、测试数据都要保持中立、可公开。

## 怎么开发

- **按 BUILD-PLAN 的 phase 走**：P0 契约与模板 → P1 核心闭环 → P2 准入与导入 → P3 迁移机制 → P4 适配器与 CLI → P5 收尾 skill。每个 phase 自成可验证的里程碑。
- **契约先行**：先把 `schemas/`（zone / skill-manifest / registry / migration）定稿，再写消费它们的 skill/CLI。
- **YAGNI + 防过度设计**：字段、机制都从最小集起步，需要再长。这套系统的灵魂之一就是"别把库搞复杂、别退化"。
- **自洽校验**：引擎自带 `examples/minimal/` 要能跑通"init→bootstrap→sync→读写→doctor 通过"。
- **迁移是一等公民**：版本升级当数据库迁移做（有序/幂等/可验证/可回滚），见 BUILD-PLAN §9。**任何迁移都不能丢数据**——git tag 快照 + doctor 前后校验 + 模糊内容进隔离区。

## 工作准则

- 先想清楚再写；不确定就说不确定 + 给选项 + 推荐。
- 外科手术式改动；与既有结构/约定一致。
- 多步任务先给计划；产出后自查占位符/矛盾/跑题。
- 改完列清楚动了哪些文件。

> 注：本仓库随附 `AGENTS.md` 是本文件的镜像（供 Codex 读）；改工作准则时两份同步。
